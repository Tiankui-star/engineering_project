## 1. 系统架构概述

### 1.1 技术栈

| 层次 | 技术选型 |
|------|----------|
| 前端 | Vue 2 + Element-UI + Axios |
| 后端框架 | SpringBoot 3 + Spring Security |
| 持久层 | MyBatis（XML 映射） + 若依代码生成基础 |
| 数据库 | MySQL 8 |
| 缓存 | Redis（Token 缓存、验证码） |
| 权限 | 基于 RBAC 的角色权限体系，三类业务角色：管理员 / 教师 / 学生 |

### 1.2 分层结构

```
Controller（HTTP接口层）
    └── Service Interface（业务接口）
            └── ServiceImpl（业务实现，含事务）
                    └── Mapper Interface（数据访问接口）
                            └── Mapper XML（SQL映射文件）
```

### 1.3 业务模块关系总览

```
department（院系）
    ├── faculty（教职工）  ← 多对一关联院系，系主任反向关联院系
    └── student（学生）    ← 多对一关联院系

faculty / student
    ├── academic_paper（论文）  ← 通过 paper_author 多对多
    ├── patent（专利）          ← 通过 patent_author 多对多
    └── research_project（科研项目）← 通过 project_member 多对多
```

## 2. 模块一：院系管理（Dep）

### 2.1 模块说明

院系是整个系统的基础组织单元，教职工和学生均归属于某一院系。院系信息由管理员维护，教职工和学生不可修改。

**数据表**：`department`

**包路径**：`com.ruoyi.Dep`

### 2.2 数据模型（Department）

| 字段名 | Java类型 | 数据库列名 | 说明 |
|--------|----------|------------|------|
| departmentId | Long | department_id | 主键，自增 |
| departmentCode | String | department_code | 院系代码，**全局唯一** |
| departmentName | String | department_name | 院系名称，**全局唯一** |
| headFacultyId | Long | head_faculty_id | 系主任教师ID（外键→faculty），**按ID唯一** |
| headFacultyName | String | ——（非DB字段） | 系主任姓名，通过 LEFT JOIN 查询展示 |
| description | String | description | 院系简介 |

> `headFacultyName` 不存储在数据库，由 Mapper 中的 LEFT JOIN faculty 动态拼接返回。

### 2.3 接口列表

| GET | `/Dep/department/{departmentId}` | `Dep:department:query` | 查询单条院系详情 |

| `updateDepartment(Department)` | 修改院系 |

| `checkDepartmentCodeUnique(Department)` | 校验院系代码是否唯一 |
| `checkDepartmentNameUnique(Department)` | 校验院系名称是否唯一 |
| `checkHeadFacultyIdUnique(Department)` | 校验系主任是否已被其他院系占用 |

### 2.5 唯一性校验实现详解

**校验时机**：Controller 的 `add()` 和 `edit()` 方法在调用 Service 保存前先逐项校验，校验失败直接 `return error(...)` 不进入数据库操作。

**校验逻辑（以院系代码为例）**：

```
ServiceImpl.checkDepartmentCodeUnique(Department department):
  1. 获取当前记录ID（新增时为-1，修改时为实际ID）
  2. 执行 SQL: SELECT department_id, department_code 
              FROM department WHERE department_code = ? LIMIT 1
  3. 若查询结果为 null → 唯一，返回 true
  4. 若查询结果.departmentId == 当前记录ID → 自身重复（修改时不报错），返回 true
  5. 否则 → 重复，返回 false
```

**系主任唯一性特殊规则**：
- 判断依据为 `headFacultyId`（教师ID），而非姓名
- 若两位教师同名但 ID 不同，视为不同人，各自担任系主任不冲突
- 若 `headFacultyId` 为 null，跳过校验（系主任可为空）

**校验失败返回的错误消息示例**：

| 场景 | 返回消息 |
|------|----------|
| 新增时代码重复 | `新增院系失败，院系代码『CS001』已存在` |
| 修改时名称重复 | `修改院系失败，院系名称『计算机学院』已存在` |
| 系主任已被占用 | `新增院系失败，该教师已是其他院系的系主任，同一人不能担任多个院系系主任` |

### 2.6 系主任级联清空机制

当教职工被删除时，需将其担任的系主任职务清空，防止孤立外键：

```
FacultyServiceImpl.deleteFacultyByFacultyIds(facultyIds):
  for each facultyId:
    departmentService.clearHeadFacultyByFacultyId(facultyId)
    → UPDATE department SET head_faculty_id = null WHERE head_faculty_id = ?
  然后执行 facultyMapper.deleteFacultyByFacultyIds(facultyIds)
```

> **权限扩展说明**：`/list` 接口使用 `hasAnyPermi`，允许论文、专利、项目模块（需选择作者时）也能调用教职工列表接口。

### 3.4 Service 方法列表

| 方法签名 | 说明 |
|----------|------|
| `selectFacultyByFacultyId(Long)` | 根据主键查询 |
| `selectFacultyList(Faculty)` | 条件查询（教师角色只查自己） |
| `insertFaculty(Faculty)` | 新增教职工，含自动创建系统用户 |
| `updateFaculty(Faculty)` | 修改教职工，含工号/姓名变更时同步 sys_user |
| `deleteFacultyByFacultyIds(Long[])` | 批量删除，含清空系主任职务和删除系统用户 |
| `deleteFacultyByFacultyId(Long)` | 单条删除 |
| `syncFacultyNumber(Long, String)` | 供外部调用的工号同步方法 |

### 3.5 新增教职工实现详解（含自动创建系统用户）

新增教职工是整个系统最复杂的写入操作之一，需保证 faculty 表与 sys_user 表的原子一致性：

```
insertFaculty(Faculty faculty) [@Transactional]:
  1. 调用私有方法 getOrCreateSysUser(faculty):
       a. 以工号作为 userName 查询 sys_user 是否已存在
       b. 若已存在 → 抛出 ServiceException("工号已存在")
       c. 同时查询 faculty 表确认工号未被占用
       d. 创建新 SysUser:
            - userName  = faculty.facultyNumber（工号）
            - nickName  = faculty.name（姓名）
            - password  = BCrypt加密("123456")（初始密码）
            - status    = "0"（正常）
            - deptId    = 100（默认部门）
            - roleIds   = [5]（默认教师角色ID）
       e. 调用 sysUserService.insertUser(newUser) 写入
       f. 重新查询获取生成的 userId 并返回
  2. 将 userId 设置为 faculty.facultyId（两表ID一致）
  3. 执行 facultyMapper.insertFaculty(faculty) 写入教职工表
```

### 3.6 修改教职工实现详解（含双向同步）

```
updateFaculty(Faculty faculty) [@Transactional]:
  1. 查询修改前的数据（existing）
  2. 判断工号（facultyNumber）是否变更：
       若变更 → 校验新工号在 sys_user 中唯一性
              → 若冲突抛出 ServiceException
  3. 判断姓名（name）是否变更
  4. 若工号或姓名有任何变更：
       构造 SysUser 对象，设置 userId、userName（工号变了）、nickName（姓名变了）
       调用 sysUserService.updateUserProfile(sysUser) 同步
  5. 调用 facultyMapper.updateFaculty(faculty) 更新教职工表
```

### 3.7 删除教职工实现详解（三步级联）

```
deleteFacultyByFacultyIds(Long[] facultyIds) [@Transactional]:
  Step 1: 遍历 facultyIds，对每个 facultyId：
          → departmentService.clearHeadFacultyByFacultyId(facultyId)
          → UPDATE department SET head_faculty_id = null WHERE head_faculty_id = ?
  Step 2: facultyMapper.deleteFacultyByFacultyIds(facultyIds)
          → DELETE FROM faculty WHERE faculty_id IN (...)
  Step 3: sysUserService.deleteUserByIds(facultyIds)
          → DELETE FROM sys_user WHERE user_id IN (...)
```

### 3.8 角色数据隔离

教职工角色（`college_teacher`）查询列表时，Service 会自动注入当前用户ID作为过滤条件：

```java
if (hasRole("college_teacher")) {
    faculty.setFacultyId(SecurityUtils.getUserId());
}
// 结果：教师只能看到自己的一条记录
```

## 5. 模块四：学术论文管理（Paper）

### 5.1 模块说明

管理学院发表的学术论文，支持多作者（教师或学生）。论文与作者通过中间表 `paper_author` 实现多对多关联。每篇论文标识一位第一作者（一作），并通过一作所属院系自动填充论文的院系归属。

**数据表**：`academic_paper`（主表）、`paper_author`（作者关联中间表）

**包路径**：`com.ruoyi.Paper`

### 5.2 数据模型（AcademicPaper）

| paperId | Long | 论文主键，自增 |
| paperTitle | String | 论文标题 |
| paperAbstract | String | 摘要 |
| journalName | String | 期刊名称 |
| volumeIssue | String | 卷期（如"Vol.12, No.3"） |
| pageNumbers | String | 页码范围 |
| publicationDate | Date | 发表日期 |
| publicationLevel | String | 发表级别（SCI/EI/北核/南核/普通） |
| doi | String | DOI号 |
| citationCount | Long | 被引次数 |
| firstAuthorType | String | 一作类型：`FACULTY`（教师）或 `STUDENT`（学生） |
| firstAuthorId | Long | 一作ID（关联 faculty_id 或 student_id） |
| firstAuthorName | String | 一作姓名（非DB字段，JOIN展示） |
| departmentId | Long | 所属院系ID（由一作院系自动填充） |
| departmentName | String | 院系名称（非DB字段，JOIN展示） |
| paperAuthorList | `List<PaperAuthor>` | 所有作者列表（非DB字段） |

### 5.3 数据模型（PaperAuthor 中间表）

### 5.6 新增论文实现详解（院系自动推断）

```
insertAcademicPaper(AcademicPaper) [@Transactional]:
  1. 判断 firstAuthorType：
       若 "FACULTY" → 查询 faculty.departmentId 并设置到 academicPaper
       若 "STUDENT" → 查询 student.departmentId 并设置到 academicPaper
  2. 执行 academicPaperMapper.insertAcademicPaper(academicPaper)
       → 论文主表写入，useGeneratedKeys 回填 paperId
  3. 若 paperAuthorList 不为空：
       为每个 PaperAuthor 设置 paperId
       批量执行 paperAuthorMapper.batchInsertPaperAuthor(authorList)
```

**院系自动推断的意义**：管理员在录入论文时无需手动选择院系，系统自动根据第一作者所属的院系填写，减少录入错误。

### 5.7 修改论文实现详解（作者先删后增）

```
updateAcademicPaper(AcademicPaper) [@Transactional]:
  1. 同样执行院系自动推断（与新增逻辑相同）
  2. 更新论文主表
  3. DELETE FROM paper_author WHERE paper_id = ?  ← 删除所有旧作者
  4. 若新 paperAuthorList 不为空 → 批量重新插入
```

这种"先全量删除、再全量插入"的策略，简化了差量更新的逻辑，适合作者数量不大的场景。

教师和学生角色只能查看自己作为作者的论文：

    academicPaper.setAuthorUserId(SecurityUtils.getUserId());

## 8. 跨模块联动与数据一致性机制

### 8.1 教职工删除级联

教职工（faculty）与多个模块存在外键依赖，删除时按序执行：

```
删除教职工:
  ① 清空其担任的院系系主任职务 (department.head_faculty_id → null)
  ② 删除 faculty 表记录
  ③ 删除对应的 sys_user 系统用户（含登录权限）
```

> 论文/专利/项目中的作者/成员记录**不级联删除**，保留历史数据。

### 8.2 工号/学号与用户名双向绑定

| 操作 | 触发方向 | 说明 |
|------|----------|------|
| 管理员修改工号 | faculty → sys_user.userName | 更新登录用户名 |
| 管理员修改姓名 | faculty → sys_user.nickName | 更新显示名称 |
| 管理员修改学号 | student → sys_user.userName | 更新登录用户名 |
| 管理员修改姓名 | student → sys_user.nickName | 更新显示名称 |

> **学生和教师无法自行修改用户名**（学号/工号），前端 Profile 页面不暴露该字段。只有管理员可以在用户管理或教职工/学生管理中修改。

### 8.3 论文/专利院系自动推断

录入论文或专利时，系统根据第一作者（一作/第一发明人）自动推断院系归属：

```
if firstAuthorType == "FACULTY":
    departmentId = faculty.departmentId  (查询 faculty 表)
if firstAuthorType == "STUDENT":
    departmentId = student.departmentId  (查询 student 表)
```

无需管理员手动选择院系，减少人工错误。

### 8.4 科研统计冗余字段维护

`faculty` 和 `student` 表中存在冗余统计字段（`paper_count`、`project_count`、`total_funding`），用于首页数据大屏展示，避免每次大屏渲染时全表聚合查询。这些字段在以下时机更新：

- 新增/删除论文时 → 更新相关作者的 `paper_count`
- 新增/删除项目时 → 更新相关成员的 `project_count`
- 项目资金变动时 → 更新负责人的 `total_funding`
